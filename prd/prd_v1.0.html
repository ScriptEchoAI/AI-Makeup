一、需求背景

1.1 现状问题

  • 用户希望在出行、年会、节假日拍照等「要出镜」的场景前，快速判断某类妆容是否适合自己，而不是仅在修图阶段「把照片变美」。
  • 现有习惯多为在小红书、抖音等平台搜索妆教，但内容无法回答「这款妆是否适合我的脸与年龄」，试错成本高。
  • 在节日给妈妈化妆等场景中，操作者常不知道从何下手（眉形、唇色浓淡、整体风格），需要关系型、代际友好的指导，而非泛化教程。

1.2 与「美妆修图类 App」的差异（定位）

维度 修图/上传类美妆 本产品（MVP）
核心任务 事后美化成片 事前决策 + 结构建议：场合/年龄/关系下的风格族与步骤级注意事项
试妆载体 常依赖上传照片叠加 实时摄像头 AR 叠妆为主（上传对比非 MVP 核心，避免与修图同质化）
输出 成片效果 可解释的「适合度 + 结构」：如眉形建议、唇色不宜过浓、浓淡档位等，后续再接跟练

二、需求目标

目标类型 描述 衡量指标（建议） 目标值（MVP 方向）
效率 用户在关键场合前，更快完成「是否尝试该类妆容」的决策，并获得可执行的细节提示。 从打开 App 到获得「结构建议 + AR 预览」的时长；任务完成率。 典型路径 ≤ 10 分钟内完成首次闭环（与需求方口头目标一致）。
体验 妆效可信，避免「强滤镜感」；在长辈举机自拍场景下，通过提示降低废片与挫败感。 主观满意度（内测问卷）、与「假脸/滤镜感」相关的负反馈率。 负反馈占比持续监控；MVP 以轻度 AR + 风格结构建议为主，不承诺高拟真。
信任 数据默认留在本地；任何云端能力需用户显式同意。 隐私相关客诉数；可选功能勾选率（未来若有云）。 默认本地保存、默认不上云；云端能力上线前须独立同意与说明。
演进 后期可加强「高还原真实妆效」；MVP 先验证 AR + 建议闭环。 版本迭代留存、功能使用率。 见「待原型确认」与 Roadmap。

三、用户与使用场景

3.1 典型用户

  • 核心受众倾向：年轻女性用户占比较高。
  • 全年龄段：产品需覆盖不同年龄段；年龄策略为「人脸估计 + 用户自选」结合，二者不一致时以用户自选为准。
  • 关系型场景：用户为长辈（如妈妈）选妆时，长辈可能举着手机完成取景——需考虑取景距离、光线等引导（见功能逻辑）。

3.2 典型场合（冷启动需采集）

首次进入需选择：场合（如旅行拍照、公司年会、节假日聚会等）、年龄段、关系（给自己化妆 / 帮长辈化妆）。具体选项列表可在原型阶段固化。

五、详细功能清单与基础逻辑

以下为 MVP 必须梳理的操作主线、前置条件、业务规则与数据流向；与界面强相关的交互细节待原型确认后写入「详细方案」章节（最终版 PRD）。

5.1 模块总览

模块优先级概要
F1 冷启动与场合画像P0首次（或重置后）必选：场合、年龄段、关系；年龄冲突以自选为准。
F2 实时 AR 试妆P0摄像头实时叠妆（轻度），强调可信预览而非厚重滤镜。
F3 风格与结构建议P0输出风格族与结构化建议：须覆盖眉形、唇色浓淡等（给妈妈妆等场景尤先）。
F4 取景辅助P0针对「长辈举机」：距离是否合适、光线是否过暗/过曝等提示（规则可迭代）。
F5 对比与本地存储P1允许偶尔截图/保存对比；默认本地，默认不上云。
F6 跟练 / 分步指导P2（形态待定）方向：checklist、短循环跟练；具体形态待原型确认后补充。

5.2 F1 冷启动与场合画像

  • 触发：首次启动；或用户在设置中「重新选择场合画像」。
  • 输入：场合（枚举，可扩展）、年龄段（自选 + 可选展示算法估计）、关系（自己 / 帮长辈）。
  • 规则：若系统估计年龄与用户选择不一致，一切以用户选择为准，系统可弱提示「已按您选择的年龄推荐」。
  • 输出：写入本次会话或用户配置（持久化策略技术定稿），驱动 F3 的建议模板与文案。

5.3 F2 实时 AR 试妆 + F4 取景辅助

  • 前置：相机权限；设备尽量覆盖多机型（需求方使用 iPhone，需在真机清单中优先包含 iOS 新机）。
  • 主线：本地视频流 → 人脸/关键点检测（实现层选型待定）→ 妆容资源叠加渲染 → 屏幕呈现。
  • 业务规则：MVP 定位为「风格与结构建议 + 轻度 AR 预览」，需在界面层面对用户预期进行管理，避免与「高拟真 AR」混淆；高拟真列为后期优化。
  • 取景辅助：当关系为「帮长辈」或检测到单人远距离自拍等策略满足时，展示距离/光线引导（具体阈值与 UI 待原型确认）。

5.4 F3 风格与结构建议（含妈妈妆）

  • 触发：完成画像且 AR 预览可用时，或用户主动刷新建议。
  • 输出结构（至少）：推荐风格族（与场合、年龄、关系关联）；注意事项列表，必须包含眉形相关提示、唇色「不宜过浓」等代际友好约束（文案可调）。
  • 数据流：画像标签 + 本地或端侧模型输出（若未来有云，须用户勾选且单独说明）→ 前端展示。

5.5 F5 对比与本地存储

  • 允许：用户偶尔截图、保存「原对比 / 叠妆对比」（具体交互待原型）。
  • 隐私：默认本地-only 存储路径；默认不上传云端；若后续提供云备份或云端精修，必须用户显式勾选并展示用途说明。

5.6 核心端到端数据流（逻辑示意）

主路径：冷启动画像 → 实时采集视频帧（本地）→ AR 渲染与建议生成（默认本地；云端为可选后续）→ 用户查看建议与预览 → 可选本地保存对比。

边界：无相机权限时，降级策略（仅文字建议或引导开启权限）待原型确认。

六、数据、隐私与合规要点

  • 默认本地保存对比与截图；默认不向云端传输用户影像用于 MVP 闭环。
  • 未来任何上传、同步、云端推理功能，需单独同意与清晰说明。
  • 对外沟通避免过度承诺「真实妆效」；MVP 与「高还原」分阶段宣传，降低客诉风险。

七、非功能需求与风险

类别说明
性能AR 预览帧率与发热需真机分级；多机型覆盖。
体验红线强烈避免「妆效假、像滤镜」—— 通过轻度叠加 + 文案 + 后续算法迭代控制。
风险人脸与年龄相关能力需注意合规与伦理表述；内容需避免医疗承诺。
